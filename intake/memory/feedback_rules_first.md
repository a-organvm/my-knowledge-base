Never reduce a universal system insight to a tactical placement decision. When the user identifies a pattern operating across scales (sorting files, sorting emails, sorting repos, naming things), the RULE SYSTEM is the work — not another ad-hoc instance with a "someday unify" note.

**Why:** The user sees the organism as one thing. Classification and routing happen at every scale (domus-sort for files, registry for repos, naming-validator for names). Making ad-hoc placement decisions (e.g., "put mail-triage in domus") perpetuates the problem — every new thing triggers the same debate. The routing law should resolve placement automatically from properties.

**How to apply:** When a placement/architecture question arises, check whether a universal rule can answer it. If not, the rule system needs extending — that's the real work, not the instance. Present the rule, show how it resolves the instance, then implement. Order: law → derivation → implementation.
