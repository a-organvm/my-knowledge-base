**What:** Conductor CLI was throwing `ModuleNotFoundError: No module named 'conductor'` on every invocation, blocking the Dispatch Protocol (`conductor_fleet_dispatch`, `conductor_fleet_guardrailed_handoff`) per CLAUDE.md.

**Root cause:** `pip show conductor` reported v0.5.1 installed editable from `/Users/4jp/Workspace/organvm-iv-taxis/tool-interaction-design`, but that directory had been deleted/renamed. The actual source had moved to `/Users/4jp/Workspace/organvm/tool-interaction-design/` (the `organvm-iv-taxis` → `organvm` rename happened upstream of the editable .pth file). Site-packages still held stale `__editable__.conductor-0.5.1.pth` and finder mapping pointing at the old location.

**Fix:** `/opt/homebrew/opt/python@3.11/bin/python3.11 -m pip install -e /Users/4jp/Workspace/organvm/tool-interaction-design/` — pip cleanly uninstalled v0.5.1 from old metadata and reinstalled editable from new location. Verified: `/opt/homebrew/bin/conductor --help` exits 0 with full subcommand list.

**Where:**
- Shim: `/opt/homebrew/bin/conductor` (unchanged, shebang `#!/opt/homebrew/opt/python@3.11/bin/python3.11`)
- Source (new): `/Users/4jp/Workspace/organvm/tool-interaction-design/conductor/`
- Editable .pth: `/opt/homebrew/lib/python3.11/site-packages/__editable__.conductor-0.5.1.pth` (regenerated by pip)

**State:** repaired 2026-04-27 — single command, no other state changes

**Related broken siblings to watch for:** Same package's pyproject also declares `mcp_server.py` and `router.py` modules. If the conductor MCP server or tool-router CLI starts failing similarly, run the same reinstall.

**Why:** Unblocks Dispatch Protocol per CLAUDE.md ("Before starting BUILD-phase work, call `conductor_fleet_dispatch`"). Without it, Claude was forced to do mechanical/tactical work that should have routed to Codex/Gemini/OpenCode, defeating the force-multiplication design.
