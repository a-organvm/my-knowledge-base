Each prompt atom is not a single work item — it's a multiverse of micro-elements. The atomization pipeline went prompts → atoms, but the next decomposition is atoms → micro-elements → implied work.

Example: "Fix the portfolio registry entry — wrong name and wrong status" contains:
- "Fix" → correction action → implies verification, prevention, regression testing
- "portfolio" → portfolio domain → implies cross-site unity, deployment, design system
- "registry entry" → registry governance → implies validation, schema enforcement
- "wrong name" → naming convention → implies naming audit, convention enforcement
- "wrong status" → promotion state machine → implies status tracking, transition validation

Each micro-element suggests work in a different universe. The true backlog is not 13,771 atoms — it's the multiverse of implications embedded within them.

**Why:** The triage and priority engines treat atoms as atomic units. But the user has repeatedly said prompts contain multi-dimensional signals. The system needs a decomposition layer that extracts micro-elements and generates the implied work surface.

**How to apply:** When processing any atom (closing, burning, reviewing), decompose it into constituent micro-elements. For each, check if corresponding coverage exists. Missing coverage = vacuum atom to be generated. This is the "3D, 40 parallel directions" principle operationalized at the sub-atomic level.
