The user's prompts operate at three levels:

1. **Macro unit** — the prompt as spoken/typed. One message.
2. **Micro units** — the discrete directives, rules, questions, signals packed inside. Average ~7 per prompt, often more.
3. **Multiverse instances** — each micro-unit implies application across MULTIPLE contexts (organs, repos, projects, systems, life domains). A directive about "naming" doesn't mean "name this one thing" — it means naming conventions must be enforced everywhere naming occurs: sessions, files, repos, plans, exports, CLI output.

**Why:** The user thinks in ideal forms and universal systems. A single prompt about "income" doesn't mean "one product needs Stripe." It means: the income problem spans consulting (padavano), products (public-record-data-scrapper, content-engine), employment (application-pipeline), grants, writing, art, and skills licensing. Each is an instance of the same micro-unit across different universes.

**How to apply:**
- When atomizing a prompt, don't stop at micro-units. For each micro-unit, enumerate the universes (organs, repos, life domains) where it applies.
- A prompt about "sequencing the ORGANVM system" has instances in every organ, every repo's seed.yaml, the registry, the governance rules, the dependency graph, and the domain architecture.
- The total directive count is micro-units × applicable universes. A 7-micro-unit prompt touching 8 organs = 56 discrete actions.
- "A little light" means: always assume there are more instances than you initially identify. The user's mental model is maximalist and recursive.
