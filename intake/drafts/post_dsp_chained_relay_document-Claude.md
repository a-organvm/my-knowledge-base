# $POST_DSP_DEEP_RESEARCH_CHAINED_RELAY_DOCUMENT

## $DOCUMENT_PURPOSE

This document converts the prior avenue-specific handoff prompts into one sequential relay. It is designed to run in a single research thread or agent context so that each stage inherits the outputs of the previous stage, updates a cumulative state object, and hands a concise inheritance package to the next stage.

The governing doctrine is unchanged.

`$CONSUMER_SHELL` names the listener-facing convenience layer.

`$COORDINATION_SUBSTRATE` names the underlying regime of catalog governance, rights, ranking, telemetry, settlement, audience ownership, and fraud control.

The research order is also unchanged.

The mandatory core remains product architecture, payout and settlement, and regulation and gatekeeping. The major differentiators remain artist-centric experiments, direct-to-fan economics, and AI, fraud, provenance, and trust. The current strategic bias also remains unchanged unless the evidence overturns it: `M5 -> M3 -> M1`, with `M4` and selected `M7` components as later substrate reforms.

---

## $RELAY_EXECUTION_PROTOCOL

### 1. $EXECUTION_MODE

Run all stages in one continuous thread.

Do not restart the research frame between stages.

Do not discard prior outputs.

Do not rewrite inherited findings unless they are revised, contradicted, or materially sharpened.

### 2. $STATE_DISCIPLINE

At the end of every stage, update the cumulative state object named `$CHAIN_STATE`.

At the end of every stage, also emit one inheritance package named `$HANDOFF_PAYLOAD_STAGE_N`.

The next stage must begin by ingesting the immediately previous handoff payload plus the cumulative state object.

### 3. $SOURCE_HIERARCHY

Use the following source order throughout the chain.

| Priority | Source class | Rule |
|---|---|---|
| 1 | Primary | Official docs, product docs, developer docs, investor materials, earnings, help centers, policy pages, legal records, regulatory filings, collection-society materials, methodology pages |
| 2 | Peer-reviewed | Journal articles, conference papers, university press work, rigorous academic analyses |
| 3 | Institutional / trade | Only when primary and peer-reviewed materials do not fully answer the question |
| 4 | Commentary | Use sparingly and only when analyzed as rhetoric rather than evidence |

### 4. $NON_NEGOTIABLE_RESEARCH_RULES

| Rule ID | Requirement |
|---|---|
| `$RULE_FACT_INFERENCE_SPLIT` | Separate observed fact from inference |
| `$RULE_DATE_DISCIPLINE` | Use exact dates for major claims and developments |
| `$RULE_SOURCE_LEDGER` | Every consequential claim must be tied to a source class and source date |
| `$RULE_RHETORIC_CONTROL` | Separate company self-description from independently supported reality |
| `$RULE_SHELL_SUBSTRATE_SPLIT` | Every stage must classify findings under `$CONSUMER_SHELL` and `$COORDINATION_SUBSTRATE` |
| `$RULE_NO_GENERIC_HISTORY` | Do not drift into generic music-industry or streaming-history filler |
| `$RULE_NO_NEUTRAL_SPOTIFY_ASSUMPTION` | Do not assume Spotify or any incumbent is the neutral baseline of truth |
| `$RULE_NO_FAIRNESS_VAGUENESS` | Do not use “fairness” as a conclusion without tracing the mechanism |
| `$RULE_MODEL_TESTING` | Continuously test the implications for `M1` through `M7`, especially `M5`, `M3`, `M1`, and `M4` |

### 5. $STANDARD_OUTPUT_OBJECTS

Every stage must update or produce the following objects.

| Object | Purpose |
|---|---|
| `$MASTER_SOURCE_REGISTER` | cumulative register of all sources used so far |
| `$MASTER_PRIMARY_FINDINGS` | cumulative fact base from primary sources |
| `$MASTER_PEER_REVIEWED_INTERPRETATION` | cumulative interpretation layer from peer-reviewed work |
| `$MASTER_CONTRADICTION_LOG` | cumulative conflicts among official claims, research, and observed mechanics |
| `$MASTER_SHELL_INVARIANTS` | preserved listener-side requirements that cannot be broken lightly |
| `$MASTER_SUBSTRATE_REDESIGN_TARGETS` | cumulative list of substrate functions that must be redesigned |
| `$MASTER_INCUMBENT_MATRIX` | evolving cross-platform matrix of incumbent structures |
| `$MASTER_MODEL_EVIDENCE_M1_TO_M7` | evidence relevant to each alternative model |
| `$MASTER_OPEN_QUESTIONS` | unresolved issues that later stages must test |
| `$MASTER_TIMELINE` | dated sequence of major platform, legal, regulatory, and market shifts |
| `$MASTER_ENTRANT_THESIS_CANDIDATES` | emerging possible post-DSP theses, revised over time |
| `$CHAIN_DELTA_STAGE_N` | what changed in the cumulative state at this stage |
| `$HANDOFF_PAYLOAD_STAGE_N` | concise inheritance package for the next stage |
